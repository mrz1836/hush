# Feature Specification: Server `/s`, `/revoke`, `/hz` Handlers + Audit Log

**Feature Branch**: `013-server-handlers-and-audit`
**Created**: 2026-04-30
**Status**: Draft
**Input**: User description: "GET /s/<name> (token-gated, scope=name + client IP enforcement, INTERACTIVE consumes uses, response body is ECIES envelope), POST /revoke (signed body, permanent revocation), GET /hz (no auth, operational state only); plus internal/audit: hash-chained + signed event log, never drops events on backpressure, optional best-effort Discord mirror"

## Overview

This chunk delivers the three remaining vault-server endpoints — secret retrieval (`/s/<name>`), explicit token revocation (`/revoke`), and health (`/hz`) — and the tamper-evident audit log that every server outcome flows through. Together they turn the server from a session-issuing front door (closed by the previous chunk) into a complete, observable secret broker.

The secret-retrieval endpoint is the throughput path: an approved client presents an issued session token, the server validates the token against the requested secret name, the requesting client's mesh IP, and (for interactive sessions) the token's remaining use-count, and on success returns the requested secret as an opaque ciphertext envelope encrypted to the client's per-session ephemeral public key. No plaintext secret value ever appears in a response body, an error path, an audit event, or an operational log record.

The revocation endpoint is the kill switch: a client that owns the registered key the token was issued to can permanently invalidate that token by token identifier within the token's remaining lifetime. The endpoint accepts a signed body, verifies the signature against the same client-key registry the claim handler uses, and refuses any request that does not prove ownership.

The health endpoint is the readiness signal for operators and monitoring within the trust mesh: it reports whether the vault is loaded, whether the chat platform is currently reachable, whether configuration is valid, and whether the clock is in sync. It is reachable without a session token (the network boundary is the trust boundary, per Constitution Principle VI) and never reveals any secret name, scope membership, registered-client identity, or token content.

The audit log is the system's tamper-evident memory. Every security-relevant outcome — the four claim outcomes from the previous chunk, every secret retrieval, every revocation, every server lifecycle event, every chat-platform connectivity transition, every reload, every permission-check failure — produces exactly one audit event. Each event carries a monotonic sequence number, a cryptographic hash that covers the previous event's hash and the current event's payload, and a signature over that hash from the audit-signing key. Tampering with any past event detectably breaks the chain at the first mutated entry. The audit writer never drops events under producer pressure: when its buffer is full, the producing handler blocks until the writer can accept the next event. An optional Discord mirror best-effort echoes events to a configured channel; mirror failures log a warning but never block, retry-storm, or affect the disk-write path.

This chunk encodes the *what* of these endpoints and the audit log — the inputs each endpoint requires, the named outcomes each endpoint produces, the redaction obligation on every response and every audit record, the audit log's chain integrity invariants, the audit writer's backpressure invariant, and the Discord mirror's best-effort discipline. It does not encode *how* these are wired internally; the choice of hash function (subject to Constitution requirements), the buffered-channel sizing, the on-disk file format, and the Discord posting strategy live in the plan phase.

Leaking the distinction into the HTTP response would let a caller probe the token store's revocation state.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Approved client fetches a secret, response body is an ECIES envelope (Priority: P1)

A client that completed an approved claim in the previous chunk presents the issued session token to the secret-retrieval endpoint, asking for one of the secret names within the token's approved scope. The handler validates the token's signature, expiry, claimed scope membership, and bound client IP against the requesting client's mesh IP, and (for an interactive session) decrements the token's remaining use-count. On success the handler reads the requested secret value from the in-memory vault, encrypts it to the client's per-session ephemeral public key (carried in the token's claims), and returns the resulting ciphertext envelope as the response body. The response body is opaque: no plaintext secret value appears in it, in any error path, in any operational log record, or in the audit event for this retrieval.

**Why this priority**: This is the throughput path of the entire product. A vault server that issues tokens but cannot deliver secrets through them is incomplete. It realizes acceptance criterion AC-2 ("vault round-trip") and AC-7 ("end-to-end ECIES, no plaintext on the wire") and is the load-bearing test of Constitution Principle III Layer 3 (ECIES end-to-end secret transport).

**Independent Test**: With an in-process token validator that returns a successfully validated set of claims for a synthesized token, an in-process vault store containing a known secret value, and an in-process ECIES encryption primitive, send a request to the handler with the matching token in the authorization header and assert: (a) the response status is `200`, (b) the response body is the ECIES envelope produced by encrypting the known secret value to the token's claimed ephemeral public key, (c) the response body does not contain the known secret value as a substring, (d) the response content type signals an opaque binary payload, (e) for an interactive token the validator's use-count decrement was called exactly once, (f) for a supervisor token the validator's use-count decrement was not called, (g) an audit event for the successful retrieval was emitted carrying the secret name but not the secret value.

**Acceptance Scenarios**:

1. **Given** an interactive session token with one remaining use, **When** an in-scope secret is requested from the bound client IP within the token's expiry, **Then** the response is HTTP `200` with an opaque ECIES envelope as the body, the token's remaining use-count is decremented, and an audit event records the retrieval with the secret name (but not the secret value).
2. **Given** a supervisor session token, **When** an in-scope secret is requested from the bound client IP within the token's expiry, **Then** the response is HTTP `200` with an opaque ECIES envelope as the body, the token's use-count is **not** decremented, and an audit event records the retrieval with the secret name (but not the secret value).
3. **Given** any successful retrieval, **When** the response body is inspected as bytes, **Then** the body does not contain the plaintext secret value as a substring (the envelope decrypts only with the ephemeral private key held by the client).

### User Story 2 — Token validation rejects out-of-scope, wrong-IP, expired, or exhausted requests (Priority: P1)

A client presents a session token to the secret-retrieval endpoint that fails one of the token-validation checks: the token's claimed scope does not include the requested secret name; or the requesting client's mesh IP does not match the IP the token was bound to at issuance; or the token has expired; or the token is an interactive session whose remaining use-count has reached zero; or the token is malformed; or the token's signature does not verify; or the token has been revoked; or the supplied token identifier was never issued. The handler returns a non-`200` status with a body that contains only a static error code and the request identifier — no token bytes, no secret name from the request, no claimed-scope listing, no client-supplied field. An audit event records the failure outcome.

**Why this priority**: These are the per-fetch enforcement obligations that keep an issued token from being repurposed. They realize AC-4 ("JWT lifecycle: rejected from a different IP, rejected after `max_uses` fetches, can be revoked, carries `session_type` in claims"). A token that an attacker captures from a compromised agent must not work from a different host, must not work after exhaustion, and must not work after revocation — and the failure responses must not leak which check rejected the request in a way that helps the attacker enumerate.

**Independent Test**: With an in-process token validator that can be configured to fail individual checks, drive the handler through each documented failure mode (out-of-scope, wrong IP, expired, exhausted interactive, revoked, unknown JTI, bad signature, malformed token) and for each assert: (a) the response status matches the documented mapping, (b) the response body contains only the static error code for the failure class and the request identifier, (c) no portion of the supplied token, the requested secret name, or the token's claimed scope appears in the response body, (d) the vault was not consulted (no call to read the secret value occurred), (e) an audit event for the failure was emitted that carries the request identifier and outcome label but not the supplied token or any byte of it.

**Acceptance Scenarios**:

1. **Given** a token whose claimed scope does not include the requested secret name, **When** the handler processes the request, **Then** the response status indicates the token cannot access this resource, the response body contains only a static error code and the request identifier, the vault is not read, and an audit event records the out-of-scope failure.
2. **Given** a token whose bound client IP does not match the requesting client's mesh IP, **When** the handler processes the request, **Then** the response status indicates an authentication failure, the response body contains only a static error code and the request identifier, the vault is not read, and an audit event records the wrong-IP failure.
3. **Given** an interactive session token whose remaining use-count is zero, **When** the handler processes the request, **Then** the response status indicates an authentication failure, no decrement is attempted, the vault is not read, and an audit event records the exhausted-token failure.
4. **Given** a token whose expiry is in the past, **When** the handler processes the request, **Then** the response status indicates an authentication failure, the vault is not read, and an audit event records the expired-token failure.
5. **Given** a token whose identifier matches a previously revoked entry, **When** the handler processes the request, **Then** the response status indicates an authentication failure, the vault is not read, and an audit event records the revoked-token failure.
6. **Given** a token whose signature does not verify, or whose JTI was never issued, or whose body is malformed, **When** the handler processes the request, **Then** the response status indicates an authentication failure, the vault is not read, and an audit event records the failure.
7. **Given** any of the above failures, **When** the response body is inspected, **Then** the body contains only a static error code and the request identifier — no token bytes, no claimed-scope listing, no requested secret name beyond the request identifier.

### User Story 3 — Token-owner revokes a session token, revocation is permanent within the token's lifetime (Priority: P1)

A client that owns the registered key the token was issued to sends a signed body to the revocation endpoint (`POST /h/<prefix>/revoke`) identifying the token by its identifier *in the request body* alongside a fresh nonce, current timestamp, and a signature over the canonical form of that body. No identifier appears in the URL path; the handler reuses the claim handler's exact canonical-JSON canonicalization and signature-verify path. On a verified signature, the handler marks the identified token as revoked. Subsequent secret-retrieval requests presenting that token are rejected by the wrong-IP/expired/exhausted check family at User Story 2, with `revoked` as the named failure cause. Revocation is permanent: a re-issued token with the same identifier (which itself is forbidden by the issuance rules) would not bypass the revocation, and re-revoking an already-revoked identifier is idempotent and audited as such.

**Why this priority**: This is the kill switch. It is part of AC-4 ("JWT lifecycle: can be revoked"). Without it, a stolen or repurposed token must wait out its TTL or its use-count before it stops working. With it, an operator who notices a problem can pull a token from the field immediately, and a daemon that detects auth drift can release its own session before exiting.

**Independent Test**: With an in-process token store that records revocation state and an in-process signature verifier configured to accept exactly one registered client key, send three revocation requests: (a) a request with a valid signature for a known, currently-active token identifier, (b) a request with an invalid signature (or a signature from a key that was not the original issuer's signer), (c) a re-revocation request for an already-revoked identifier. Assert: (a) returns success status and the store records the token as revoked; (b) returns the documented failure status and the store is unchanged; (c) returns the documented success/idempotent status and the store is unchanged from its already-revoked state. Assert that an audit event is emitted for each request, and that no event contains the token's bytes, the supplied signature, or the request body's nonce.

**Acceptance Scenarios**:

1. **Given** a request body that is well-formed and signed by the registered client key the token was issued to, **When** the handler processes it, **Then** the response indicates success, the token store records the identifier as revoked, an audit event records the revocation outcome, and any subsequent secret-retrieval request bearing that token is refused on the basis of revocation.
2. **Given** a request body whose signature does not verify against a registered client key (or signs over a canonical form that does not match the supplied body), **When** the handler processes it, **Then** the response indicates a signature failure with a static error code and the request identifier, the token store is unchanged, and an audit event records the failure.
3. **Given** a re-revocation request for an identifier that is already recorded as revoked, **When** the handler processes it, **Then** the response is HTTP `200` with the *same* static body (success code + request identifier) returned for a first revocation, the token store remains in the revoked state, and an audit event records the idempotent re-revocation under a distinct outcome label (`revoke_idempotent_already_revoked`) in the chain.
4. **Given** any revocation response (success or failure), **When** the response body is inspected, **Then** the body does not contain the supplied signature, the supplied nonce, the token's bytes, or any field beyond the static outcome code and the request identifier.

### User Story 4 — Health endpoint reports operational state without a session token, and never reveals a secret name or token content (Priority: P1)

An operator or a downstream monitoring agent within the trust mesh sends an unauthenticated request to the health endpoint. The handler returns a JSON body describing the server's operational state: a constant status indicator confirming the server is responding, time-since-startup (`uptime`), the count of secrets currently loaded in the vault (`secrets_count` — never the names), the count of currently-valid session tokens (`active_tokens` — never the identifiers), whether the chat-platform connection is currently usable for new approvals (`discord_connected`), whether configuration is valid (`config_valid`), and whether the system clock is sufficiently in sync (`clock_in_sync`). The body never carries any secret name, any registered-client identifier, any token identifier or its claims, the random API path prefix beyond the path the request itself used, any token content, the audit chain's most recent hash, the audit signing key, the chat-platform bot token, or any other secret material.

**Why this priority**: The health endpoint realizes acceptance criterion AC-1 ("`hush serve` produces a running vault server that responds to `GET /h/{prefix}/hz`") and is the operator's first signal during startup, network outage, or reload. It must be reachable without a session token (Constitution Principle VI: the Tailscale mesh is the auth perimeter for this signal) and must be aggressively redacted: it is the easiest endpoint to accidentally turn into an enumeration vector for secret names or registered clients.

**Independent Test**: With an in-process server harness, a vault loaded with a known set of secret names, and a chat-platform stub that can be flipped between connected and disconnected, send unauthenticated requests to the health endpoint and assert: (a) the response is `200` regardless of authentication, (b) the body parses as JSON with the documented operational fields (`status`, `uptime`, `secrets_count`, `active_tokens`, `discord_connected`, `config_valid`, `clock_in_sync`), (c) `secrets_count` equals the harness's loaded-secret count and `active_tokens` equals the count of non-revoked non-expired tokens in the harness store, (d) the body's `discord_connected` value matches the stub's flipped state, (e) the body does not contain any of the loaded secret names as a substring (sentinel-leak with names containing a unique sentinel), (f) the body does not contain any registered-client identifier, any token identifier from the harness store, the audit-signing key, or the chat-platform bot token as a substring.

**Acceptance Scenarios**:

1. **Given** a request to the health endpoint with no session token in the authorization header, **When** the handler processes it, **Then** the response is HTTP `200` with a JSON body carrying `status`, `uptime`, `secrets_count`, `active_tokens`, `discord_connected`, `config_valid`, and `clock_in_sync`.
2. **Given** the chat-platform connection is flipped between connected and disconnected, **When** the health endpoint is queried in each state, **Then** the body's `discord_connected` field reflects the current state.
3. **Given** the vault is loaded with a set of secret names containing a sentinel string, **When** the health endpoint is queried, **Then** the response body's `secrets_count` equals the loaded count and the sentinel string does not appear anywhere in the body.
4. **Given** the registered-client registry, the audit-signing key, the chat-platform bot token, and any token identifier in the active-token store contain unique sentinel substrings, **When** the health endpoint is queried, **Then** the response body does not contain any of those sentinels (the body carries counts, not identifiers).

### User Story 5 — Every server outcome appends exactly one event to the hash-chained, signed audit log (Priority: P1)

Every outcome the server produces — every claim outcome from the previous chunk, every secret retrieval (success or failure), every revocation (success or failure), every server lifecycle transition (start, stop, reload), every chat-platform connectivity transition (disconnect, reconnect), every permission-check failure — produces exactly one audit event. Each event carries a monotonically increasing sequence number, the timestamp at which the writer accepted it, an action label, a structured data payload that includes the request identifier and any relevant scope/IP/secret-name/JTI/outcome metadata (but never any secret value, any signature byte, any token byte, or any chat-platform bot-token byte), the cryptographic hash of the previous event's hash concatenated with the current event's payload, and a signature over the current event's hash produced by the audit-signing key. The first event in the chain links to a fixed, well-defined genesis hash so the chain has a verifiable starting point. Verifying the chain end-to-end requires only the audit-signing public key and the genesis hash.

**Why this priority**: The audit chain is Constitution Principle III Layer 6 — the system's tamper-evident memory. Without it, a successful intrusion can edit the disk to rewrite history. With it, a successful intrusion can delete entries (which is detectable as a sequence-number gap or as a chain break) or append new entries (which fails signature verification because the audit-signing key is in the server's process memory only) but cannot silently rewrite the past.

**Independent Test**: With an in-process audit writer, an in-process audit-signing key, and a configurable disk path, drive the writer through a known sequence of outcomes (for example: server start, claim approved, secret retrieved, secret retrieval failed for wrong IP, revoke succeeded, claim denied, server stop), close the writer, and verify the written chain end-to-end: (a) sequence numbers are contiguous from `1` and monotonically increasing, (b) each event's previous-hash field equals the previous event's hash, (c) each event's signature verifies against the audit-signing public key over the event's hash, (d) tampering with any one event's data field and re-running verification surfaces the chain-break sentinel error at the first tampered event, (e) deleting an event from the middle of the file and re-running verification surfaces the chain-break sentinel error at the gap.

**Acceptance Scenarios**:

1. **Given** a sequence of `N` outcomes driven through the audit writer, **When** the chain is verified end-to-end, **Then** there are exactly `N` events with sequence numbers `1..N`, every previous-hash linkage holds, and every signature verifies against the audit-signing public key.
2. **Given** an audit event recording a secret retrieval, **When** the event's data payload is inspected, **Then** the payload contains the secret name and the request identifier and never the secret value as a substring.
3. **Given** any audit event recording a token-validation failure, **When** the event's data payload is inspected, **Then** the payload contains the request identifier, the failure-class label, and any non-secret metadata about the request, but never any byte of the supplied token.
4. **Given** the audit log file on disk, **When** any single event's data field is mutated (for example a single byte flipped) and the chain is re-verified, **Then** the verification surfaces a chain-break sentinel error and the surfaced error identifies the first event whose linkage no longer holds.
5. **Given** the audit log file on disk, **When** any one event is deleted from the middle of the file and the chain is re-verified, **Then** the verification surfaces a chain-break sentinel error at the position of the gap.
6. **Given** the audit log file on disk, **When** any new event is appended whose signature was not produced by the audit-signing key, **Then** the verification surfaces a signature-failure sentinel error at that event.

### User Story 6 — Audit writer blocks the producing handler under buffer pressure rather than dropping events (Priority: P1)

A burst of concurrent requests floods the server faster than the audit writer can persist events. The writer's internal buffer fills. Subsequent producers (handlers attempting to append events) block on the buffer until the writer drains. The writer never silently drops, coalesces, or reorders events. The throughput cap is therefore visible to operators as latency under burst load, not as silent gaps in the audit record.

**Why this priority**: Backpressure-by-blocking is the only way to keep the chain complete under load. A writer that drops events on a full buffer would create silent audit gaps — which is functionally indistinguishable from a successful tamper. Operators cannot tell "we never recorded that retrieval" apart from "an attacker erased that retrieval" if drops are allowed. Constitution Principle III Layer 6 requires tamper-evidence; tamper-evidence requires that every outcome reach the chain.

**Independent Test**: With an in-process audit writer whose buffer size is set to a small known value and whose persistence step can be paused on demand, pause persistence, append a number of events larger than the buffer in close succession from concurrent producers, observe that producers attempting to append after the buffer is full do not return until persistence is resumed, resume persistence, and assert that exactly the produced number of events appear in the chain in the order they were submitted (after accounting for whatever ordering guarantee the writer commits to among concurrent producers, which must be at minimum that no event is lost).

**Acceptance Scenarios**:

1. **Given** an audit writer with buffer size `B` and `B+K` concurrent append requests for `K > 0`, **When** the writer's persistence step is paused, **Then** at most `B` appends complete and the remaining `K` block; when persistence is resumed, the remaining `K` complete and exactly `B+K` events appear in the chain.
2. **Given** any volume of concurrent producers, **When** the writer is run to completion, **Then** the number of events in the chain equals the number of `Append` calls that returned without an error.
3. **Given** the writer is operating under sustained load, **When** any single `Append` returns success, **Then** the event has been incorporated into the chain (its sequence number is final and its predecessor links are determined) before the call returns.

### User Story 7 — Discord audit mirror is best-effort and never blocks the on-disk audit chain (Priority: P2)

When the server is configured with a chat-platform audit channel, the audit writer also publishes each event to that channel as a human-readable summary. Mirroring is best-effort: if the chat platform is unreachable, slow, rate-limiting, or returns an error for a single event, the writer logs a warning describing the failure (and only the non-secret metadata of the failed event), and proceeds to the next event without retrying indefinitely. The on-disk audit chain is unaffected: every event that the writer accepted reached disk regardless of whether its mirror copy succeeded. Mirror failures never block, never re-enqueue events into the writer's buffer, never inflate latency for the producing handler, and never cause the writer to drop or reorder on-disk events.

**Why this priority**: The chat-platform mirror is a convenience for operator visibility; the on-disk chain is the source of truth. A mirror that could block the writer would convert a chat-platform outage into an audit-chain outage and (per User Story 6) into a server-handler outage. A mirror that could re-enqueue would create event duplication on the chain. Both regressions defeat the design intent of Layer 6 audit.

**Independent Test**: With an in-process audit writer wired to an in-process Discord-mirror stub whose `Publish` call can be configured to succeed, fail immediately, fail slowly, or return an error, drive the writer through a sequence of events under each mirror configuration and assert: (a) the on-disk chain is complete and consistent in every configuration, (b) the handler's `Append` latency does not grow with mirror latency, (c) a mirror failure produces a warning log record naming the failed event by sequence number and outcome (not by signature, token, or secret value), (d) the writer does not retry the same event repeatedly when the mirror fails, (e) the writer does not insert a duplicate or compensating event into the chain in response to a mirror failure.

**Acceptance Scenarios**:

1. **Given** the chat-platform mirror is unreachable, **When** events are appended, **Then** every event reaches the on-disk chain and a warning log record is emitted for each mirror failure (or once per outage burst, at the writer's discretion), and the producing handler observes no additional latency.
2. **Given** the chat-platform mirror is configured but the mirror channel identifier is empty, **When** the writer runs, **Then** mirroring is disabled, no mirror calls are made, and the on-disk chain is written normally.
3. **Given** any mirror failure, **When** the warning log record is inspected, **Then** it does not contain any secret value, any byte of any signature, any token bytes, or the chat-platform bot token.

### User Story 8 — Audit writer drains its buffer cleanly on shutdown and never abandons in-flight events (Priority: P2)

When the server begins shutdown — graceful stop, SIGTERM, context cancellation — the audit writer stops accepting new `Append` calls (those calls return a documented shutdown error), drains the events already buffered, persists each one with its hash and signature in order, and then returns from its run loop. The on-disk chain at the end of shutdown contains every event whose `Append` call returned success.

**Why this priority**: The audit chain's value depends on the invariant that "if `Append` returned success then the event reached the chain." If shutdown could discard buffered-but-not-yet-persisted events, that invariant would be broken at the most operationally critical moment (the moment something went wrong enough to trigger shutdown). The crash-vs-graceful-shutdown distinction is real (a crash may lose in-flight events; that is documented as residual risk), but a graceful shutdown must drain.

**Independent Test**: With an in-process audit writer whose buffer is partially full, request graceful shutdown and assert: (a) any in-flight `Append` call that already started before shutdown completes successfully or with a documented shutdown error consistent with whether the event reached the chain, (b) every event whose `Append` returned success appears in the on-disk chain after shutdown returns, (c) no new `Append` call started after shutdown returns success — they all return the documented shutdown error, (d) the writer's run loop returns within a bounded time after the last buffered event is persisted.

**Acceptance Scenarios**:

1. **Given** the writer has `K` events in its buffer and shutdown is requested, **When** shutdown completes, **Then** all `K` events are present on disk with valid hash linkage and signatures, and the writer's run loop has returned.
2. **Given** shutdown has been requested, **When** a new `Append` is invoked, **Then** the call returns a documented shutdown error and no new event is added to the chain.

### Edge Cases

- **Authorization header is absent or malformed for `/s`.** The handler returns HTTP `401` with a body containing only a static error code and the request identifier; the vault is not read; an audit event records the missing-or-malformed-token failure.
- **Authorization scheme is something other than the documented bearer token.** The handler treats it as a malformed-token failure (same status, same static error code) — it does not enumerate which schemes are supported through error-message variation.
- **The requested secret name does not exist in the vault but is within the token's claimed scope.** The handler returns HTTP `404` with a body containing only a static error code and the request identifier; an audit event records the missing-secret outcome with the requested name (since the name was already in the token's scope, it is not new information for an attacker). The handler does not reveal whether the same name is or is not in any other token's scope.
- **The requested secret name does not exist in the vault and is also not in the token's claimed scope.** The handler returns the out-of-scope failure (User Story 2) — the existence of the name is not disclosed.
- **An interactive token's use-count decrement and the actual vault read are interleaved with concurrent fetches.** Decrement is atomic and ordered such that the count never goes negative and never delivers more uses than the original cap. If two concurrent requests for the same token compete for the last remaining use, exactly one succeeds and the other is rejected as exhausted.
- **The vault is reloaded mid-request.** A request that was in progress when SIGHUP arrived completes against the vault state it observed; subsequent requests use the new vault. A retrieval that hits a secret that was rotated produces the new value (or the missing-secret outcome if the rotation removed it). The audit event records the secret name, not the value, so the rotation does not create a value-leak path.
- **`/revoke` request body is malformed JSON or missing required fields.** The handler returns HTTP `400` with a body containing only a static error code and the request identifier; the token store is unchanged; an audit event records the failure.
- **`/revoke` request identifies a token-id that was never issued.** The handler treats this as a signature failure rather than a distinct "unknown JTI" status — the response code and static error code are the same as for a genuine signature failure, so the endpoint cannot be used to enumerate which token identifiers exist.
- **`/hz` is queried before the vault is loaded (during early startup).** The handler returns `200` with the body's "vault loaded" field set to false; it does not return `503` or refuse the request, because doing so would gate operator visibility precisely when it is most needed.
- **`/hz` is queried while the chat-platform connection is in transition (connecting, reconnecting after a disconnect).** The "chat-platform connected" field reflects the current observable state; it is not a prediction. A request that arrives during a transition observes whichever state is current at that instant.
- **The audit log file is missing or empty at startup.** The writer initializes the chain with the genesis predecessor hash and emits a `server_start` event as sequence number `1`. No prior data is invented.
- **The audit log file exists but its last event's hash does not link to the previous event.** Startup detects the chain break and emits a Critical-tier alert; the writer continues to append from the existing tail, since refusing to record going forward would compound the loss. The existing break is not retroactively repaired.
- **Disk write to the audit log file fails (file system full, permissions error).** The producing handler's `Append` call returns the failure error; the handler's response to the client is the documented unavailable mapping (`503` for the claim handler, equivalent for the secret handler — see FR-031); a Critical-tier alert is raised. The writer does not silently discard the event.
- **The chat-platform mirror is enabled but the channel ID is misconfigured (channel does not exist or the bot lacks access).** The first publish attempt logs a warning (with the channel ID and the underlying error class, never the bot token), and subsequent attempts may be rate-limited by the writer's mirror throttle to avoid log flood. The on-disk chain is unaffected.
- **Two concurrent revocations target the same token identifier.** Both succeed (the operation is idempotent); the token store's terminal state is "revoked"; two audit events are emitted (one for each successfully verified request).

## Requirements *(mandatory)*

### Functional Requirements

#### Secret retrieval (`/s/<name>`)

- **FR-001**: The handler MUST require a session token in the request's authorization header; an absent, malformed, unsupported-scheme, or unparseable token MUST be rejected with HTTP `401` (the same status as every other token-validation failure family — see FR-009).
- **FR-002**: The handler MUST validate the supplied session token's signature, expiry, claimed scope membership against the requested secret name, bound client IP against the requesting client's mesh IP, revocation status, and (for an interactive session) remaining use-count, before reading any secret material.
- **FR-003**: For an interactive session token, the handler MUST atomically decrement the token's remaining use-count exactly once per accepted retrieval. The decrement MUST be ordered such that no token ever delivers more retrievals than its original cap. The decrement MUST NOT occur for a supervisor session token.
- **FR-004**: On a fully validated request, the handler MUST read the requested secret value from the in-memory vault, encrypt it under the token's claimed ephemeral public key using the project's ECIES primitive, and return the resulting ciphertext envelope as the response body with an opaque-binary content type.
- **FR-005**: The response body for a successful retrieval MUST be the ECIES envelope and nothing else (no JSON wrapping, no metadata fields, no trailing or leading bytes); the response MUST NOT include the secret value as a substring of any header, body, or trailer.
- **FR-006**: When the requested secret name does not exist in the vault but is within the token's claimed scope, the handler MUST return HTTP `404` with a body containing only a static error code and the request identifier.
- **FR-007**: When the requested secret name is not within the token's claimed scope, the handler MUST return HTTP `403` with a body containing only a static error code and the request identifier; the handler MUST NOT distinguish "not in scope and exists" from "not in scope and does not exist".
- **FR-008**: When the bound client IP does not match the requesting client's mesh IP, the handler MUST return HTTP `401` without consulting the vault.
- **FR-009**: When the token has expired, has been revoked, has an unknown identifier, has been exhausted (interactive only), has a malformed structure, has a signature that does not verify, is absent from the request, or is present under an unsupported authorization scheme, the handler MUST return HTTP `401` without consulting the vault. The same `401` is used for every member of this failure family — the response status MUST NOT distinguish among these failure causes; only the static error-code body distinguishes failure classes for legitimate operator/test inspection.

#### Revocation (`/revoke`)

- **FR-010**: The handler MUST be mounted at `POST /h/<prefix>/revoke` (no URL-path component carries the token identifier). The handler MUST require a request body that proves ownership of the registered client key the token was issued to. The proof of ownership MUST be a signature over the same canonical-JSON form of the request body that the claim handler uses for its own request signatures, verified against the same client-key registry the claim handler uses; the canonicalization and verification routines MUST be the identical implementation as `/claim`'s.
- **FR-011**: The signed body MUST identify the token to revoke by its identifier (a body field, not a URL path component) and MUST include the same replay-protection fields the claim handler requires (a fresh nonce within the configured replay window and a timestamp within the configured ±freshness window).
- **FR-012**: On a verified request, the handler MUST mark the identified token as revoked. The revocation state MUST persist for at least the remainder of the token's lifetime, such that any subsequent secret-retrieval request bearing that token is refused on the basis of revocation.
- **FR-013**: A revocation request whose signature does not verify, whose nonce was already seen within the replay window, whose timestamp is outside the freshness window, or whose body is malformed MUST be refused with the documented failure status; the token store MUST remain unchanged.
- **FR-014**: A revocation request for an identifier that is already recorded as revoked MUST succeed idempotently. The HTTP response MUST be identical to a first-time successful revocation (HTTP `200`, same static body containing only the success code and the request identifier — no field that distinguishes "already revoked" from "newly revoked"). The token store's state MUST remain "revoked". An audit event MUST record the idempotent re-revocation under a distinct outcome label (e.g. `revoke_idempotent_already_revoked`); the first/idempotent distinction lives only in the audit chain, never in the HTTP response.
- **FR-015**: A revocation request for an identifier that was never issued MUST be refused with the same status and static error code as a genuine signature failure; the response MUST NOT distinguish "unknown identifier" from "bad signature" in a way that lets a caller enumerate issued identifiers.
- **FR-016**: The handler MUST NOT include in any response body the supplied signature, the supplied nonce, the token identifier, or any field beyond a static outcome code and the request identifier.

#### Health (`/hz`)

- **FR-017**: The handler MUST be reachable without a session token; the network boundary (Tailscale mesh) is the auth perimeter for this signal (Constitution Principle VI).
- **FR-018**: The response body MUST be a JSON document carrying these operational fields: a constant `status` string ("ok") indicating the server is responding, an `uptime` value reporting time-since-startup, a numeric `secrets_count` (the count of secrets currently loaded in the vault — never the names), a numeric `active_tokens` (the count of currently-valid, non-revoked, non-expired session tokens — never any token identifier or its claims), a `discord_connected` boolean, a `config_valid` boolean, and a `clock_in_sync` status (either a boolean or a small enumerated state).
- **FR-019**: The response body MUST NOT contain any secret name, any registered-client identifier, any token identifier or its claims, the chat-platform bot token, the audit signing key, the audit chain's hash state, or any other secret material. Numeric counts (`secrets_count`, `active_tokens`) are permitted; identifier-bearing fields are not.
- **FR-020**: The response body MUST NOT echo the random API path prefix beyond what is implicit in the URL the request itself used; the body MUST NOT serve as a canonical source for the prefix.
- **FR-021**: The handler MUST return `200` regardless of operational state (a vault not yet loaded is reported as `vault_loaded=false`, not as a non-`200` status); the response status MUST NOT itself encode operational health beyond "the server is responding".
- **FR-021a**: The `/hz` handler MUST NOT emit audit events. Health probes are non-security-relevant operational signals; recording them would dilute the audit chain and amplify producer backpressure under monitor-agent poll rates. The User Story 5 / FR-027 enumeration is the closed set of audited outcomes.

#### Audit log — chain integrity

- **FR-022**: Every accepted audit event MUST carry a monotonically increasing sequence number, starting at `1` for the first event after a fresh log initialization.
- **FR-023**: Every event MUST carry the cryptographic hash of (the previous event's hash || the canonical form of the current event's payload), where the previous-hash for the first event is a fixed, well-defined genesis value documented in the project security documentation.
- **FR-024**: Every event MUST carry a signature over the event's hash produced by the audit-signing key (Constitution Principle III: the audit-signing key is derived from the BIP32 hierarchy and exists only in the server's process memory).
- **FR-025**: The audit chain MUST be verifiable end-to-end given only the audit-signing public key and the genesis hash: any tampered, deleted, or maliciously appended event MUST surface the documented chain-break sentinel error at the first inconsistent event.
- **FR-026**: The canonical form used for hashing MUST be the same canonical-JSON form used by the request-signing primitive, so that audit events are reproducibly hashable across implementations.

#### Audit log — content discipline

- **FR-027**: Every audit event's data payload MUST include the request identifier (when the event corresponds to a request-driven outcome) and an outcome label drawn from a fixed enumeration that covers the chunk's outcomes (server lifecycle, claim outcomes from the previous chunk, secret retrieval, revocation, vault reload, permission-check failure, chat-platform connectivity transitions, audit-mirror failures).
- **FR-028**: An audit event recording a secret retrieval MUST include the secret name; it MUST NOT include the secret value or any byte of the secret value.
- **FR-029**: An audit event recording a token-validation failure or a revocation MUST include the failure-class label and the request identifier; it MUST NOT include the supplied token, the supplied signature, the request body's nonce, or any byte of any of those values.
- **FR-030**: No audit event MUST contain the chat-platform bot token, the audit-signing key, the vault encryption key, or any other secret material — even when the event records an event that involved that material (for example, a chat-platform reconnect event records the connectivity transition, not the token used).

#### Audit log — backpressure invariant

- **FR-031**: When the audit writer's buffer is full, the producing handler's `Append` call MUST block until the writer can accept the event; the writer MUST NOT silently drop, coalesce, or reorder events. A producing handler whose `Append` call cannot be served because the writer is shutting down MUST receive a documented shutdown error and propagate it to its caller's response (mapping to the documented unavailable status for that handler).
- **FR-032**: The audit writer MUST be implemented such that events are accepted into the buffer in the order their `Append` calls returned, persisted in that order, and assigned sequence numbers in that order. If the writer permits multiple producers to append concurrently, the ordering relation between two concurrent `Append` calls MUST still be a total order whose sequence numbers are determined by the writer at acceptance time and visible in the on-disk chain.
- **FR-033**: An `Append` call that returns success MUST guarantee that the event has been incorporated into the on-disk chain (its sequence number is final, its predecessor linkage is determined, its signature is present); it is acceptable for the on-disk persistence flush to lag the call return, but the event's identity in the chain MUST be final at return.

#### Audit log — Discord mirror discipline

- **FR-034**: When the chat-platform audit mirror is enabled by configuration, the writer MUST publish each accepted event to the configured channel as a human-readable summary in addition to the on-disk chain.
- **FR-035**: Mirror publication MUST be best-effort: a failure (network error, rate limit, channel error, transport disconnect) MUST log a warning describing the failure (sequence number and outcome label only — never the signature, the token, the secret value, or the bot token) and MUST NOT block, retry indefinitely, or otherwise affect the on-disk chain.
- **FR-036**: When the mirror is configured but its channel identifier is empty, the writer MUST disable mirroring entirely (no mirror calls) and proceed with on-disk persistence as normal.
- **FR-037**: A mirror failure MUST NOT cause the writer to insert a compensating event into the on-disk chain or to retry the same event in a way that would result in chain duplication.

#### Audit log — lifecycle

- **FR-038**: The audit writer MUST be implemented such that the on-disk file handle and the sequence-counter state have a single owner; producers MUST interact with the writer only through the documented `Append` contract and MUST NOT share writer-internal state with one another.
- **FR-039**: On graceful shutdown (request-context cancellation, SIGTERM, or equivalent), the writer MUST stop accepting new `Append` calls (returning the documented shutdown error), drain its buffer to disk, and return from its run loop within a bounded time. The shutdown drain MUST preserve every event whose `Append` call returned success.
- **FR-040**: On startup, when the audit log file exists, the writer MUST initialize its sequence-number counter and previous-hash linkage from the file's last event; when the file does not exist, the writer MUST initialize from the genesis predecessor hash and start sequence numbers at `1`.

#### Operational logging

- **FR-041**: Operational log records emitted by the secret-retrieval handler, the revocation handler, the health handler, and the audit writer MUST never contain any secret value, any token bytes, any audit-signing key bytes, or the chat-platform bot token.
- **FR-042**: Operational log records MUST NOT duplicate audit content (Constitution Principle X). When an outcome is recorded in the audit chain, the operational log records the same outcome at debug-level only, with the sequence number and outcome label as the correlation handle (never the audit event's full signed payload).

#### Constitutional guarantees (verbatim)

- **FR-043**: There MUST NOT exist any code path through the secret-retrieval handler that returns a `200` response without first having validated the supplied session token end-to-end (signature, expiry, scope, IP, revocation, use-count for interactive). A token-validation failure MUST NOT result in a successful retrieval.
- **FR-044**: There MUST NOT exist any code path through the revocation handler that marks a token as revoked without first having verified the request's signature against the registered client-key registry. A signature failure MUST NOT result in a successful revocation.
- **FR-045**: There MUST NOT exist any configuration knob, environment variable, command-line flag, build tag, or runtime mode that causes the audit writer to drop events on backpressure (FR-031), to publish secret values to the chat-platform mirror (FR-035), or to weaken any chain-integrity invariant (FR-022 through FR-026).

### Key Entities

- **Session token**: The short-lived bearer credential issued by the claim handler. Carries claims for scope, bound client IP, expiry, session type, ephemeral public key, identifier, and (for interactive) remaining use-count. Validated by the secret-retrieval handler on every request and revoked permanently by the revocation handler on demand.
- **ECIES envelope**: The opaque ciphertext returned as the body of a successful secret retrieval. Decrypts only with the ephemeral private key the client retained from its claim. The plaintext value never appears in this envelope's serialised form, in any handler error path, or in any log record.
- **Revocation request**: The signed body sent to the revocation endpoint. Carries the token identifier to revoke, a fresh nonce, a current timestamp, and a signature over the canonical form of the body produced with the registered client key.
- **Health snapshot**: The JSON document returned by the health endpoint. Carries vault-loaded, chat-platform-connected, configuration-valid, and clock-sync fields and never any secret material.
- **Audit event**: The hash-chained, signed unit of the audit log. Carries sequence number, timestamp, action label, structured data payload, previous-hash linkage, current-event hash, and signature.
- **Audit chain**: The ordered, hash-linked, signed sequence of audit events on disk. Verifiable end-to-end with the audit-signing public key and the documented genesis predecessor hash.
- **Audit writer**: The single-goroutine owner of the audit chain's buffered channel and on-disk file handle. Accepts events through the `Append` contract, persists them in order, blocks producers under buffer pressure, and drains cleanly on shutdown.
- **Discord audit mirror**: The optional best-effort publisher that echoes accepted audit events to a configured chat-platform channel as human-readable summaries; a non-load-bearing convenience for operator visibility.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of successful secret-retrieval response bodies are ECIES envelopes that decrypt to the requested secret value with the correct ephemeral private key and fail to decrypt with any other key — measured by a round-trip test that encrypts a known secret to a generated ephemeral public key, decrypts the response body with the corresponding private key, and asserts equality with the original; and asserts decryption with any other key fails with the documented error.
- **SC-002**: 100% of secret-retrieval response bodies (success or error) contain no plaintext secret value as a substring — measured by a sentinel-leak test where every secret in the test vault contains a unique sentinel string and the suite captures every response body for every documented outcome and asserts no sentinel appears.
- **SC-003**: 100% of token-validation failures produce a non-`200` response, no vault read, no use-count decrement, and exactly one audit event recording the failure class — measured by exhaustively driving the secret-retrieval handler through every documented validation-failure mode (out-of-scope, wrong IP, expired, exhausted, revoked, malformed, unknown JTI, bad signature).
- **SC-004**: 100% of revocation requests with an unverifiable signature, replayed nonce, stale timestamp, or unknown token identifier produce the same documented failure status and static error code, such that no caller can distinguish an unknown identifier from a signature failure — measured by a test that captures the response body and status for each failure class and asserts the status and error code are identical for the unknown-identifier and bad-signature cases.
- **SC-005**: 100% of health-endpoint responses are `200` regardless of the vault's load state, the chat-platform connectivity, or the configuration-valid state — measured by driving the harness through every combination of those operational states and asserting the response status is always `200` and the body's fields reflect the harness state.
- **SC-006**: 100% of health-endpoint response bodies contain none of: any secret name from the loaded vault, any registered-client identifier, any token identifier from the active-token store, the chat-platform bot token, the audit signing key, the audit chain's most recent hash — measured by a sentinel-leak test where each of those values is set to a unique sentinel and the body is searched. Numeric counts (`secrets_count`, `active_tokens`) are permitted and asserted to match the harness state.
- **SC-007**: 100% of audit chains produced by the writer end-to-end satisfy: contiguous monotonic sequence numbers from `1`, valid previous-hash linkage at every event, valid signature at every event under the audit-signing public key — measured by an end-to-end verification test that runs after every documented outcome in the suite.
- **SC-008**: A tampered audit event surfaces the chain-break sentinel error at the first inconsistent event in 100% of cases — measured by a test that mutates a single byte at every offset of every event's data payload, runs verification, and asserts the surfaced error names the mutated event.
- **SC-009**: When the audit writer's buffer is full, every producer's `Append` call either blocks until the buffer drains or returns the documented shutdown error; in 0% of runs does an `Append` call return success and the event subsequently fail to appear on disk — measured by a stress test that fills the buffer, observes blocking, drains, and asserts every successful `Append` is reflected on disk.
- **SC-010**: Across `N` concurrent producers each appending `M` events with race detection enabled, exactly `N × M` events appear on the chain with monotonically increasing sequence numbers and valid linkage; the test runs race-clean with no goroutine leaks — measured by the project's standard race-clean concurrency test pattern.
- **SC-011**: A chat-platform mirror failure produces a warning log record that does not contain any secret value, any token byte, any signature byte, or the chat-platform bot token, and does not block the writer's on-disk persistence path or inflate the producing handler's `Append` latency beyond the on-disk persistence cost — measured by a test that toggles the mirror stub through immediate-failure, slow-failure, and rate-limited-failure modes and captures both the on-disk chain and the producing handler's wall-clock latency.
- **SC-012**: A graceful shutdown initiated while the writer's buffer contains `K` events results in `K` events being persisted to the chain before shutdown returns; in 0% of runs does a graceful shutdown leave a successful-`Append` event off the chain — measured by a test that pauses the persistence step, fills the buffer, requests shutdown, resumes persistence, and asserts the chain length matches the successful-`Append` count.
- **SC-013**: Test coverage of `internal/server` handlers (the chunk's three new handlers) is at least 95%; test coverage of `internal/audit` is 100% — measured by the project's test-coverage tooling against the package coverage policy in Constitution Principle VIII.
- **SC-014**: A retrieval request whose response would have included a secret value but is forced to fail at any post-validation step (vault read error, ECIES encrypt error, response write error) produces an error response whose body does not contain the secret value as a substring — measured by a sentinel-leak test that injects a sentinel as the secret value, forces each post-validation failure, and greps the captured response and log output.
- **SC-015**: Across all server endpoints owned by this chunk, no operational log record produced during a successful or unsuccessful request contains any secret value, the supplied session token, the chat-platform bot token, the audit-signing key, or the audit chain's most recent hash — measured by a sentinel-leak test that injects unique sentinels for each of those secrets and asserts none appear in the captured log output.

## Assumptions

- The session-token validate/store/revoke primitive is owned by SDD-07 and is available to this chunk through the server's already-established wiring. The handler depends on, but does not implement, the token primitives.
- The ECIES encryption primitive is owned by SDD-09 and is available as a function the secret-retrieval handler can call to produce the response envelope. This chunk depends on, but does not implement, ECIES.
- The canonical-JSON form, request-signature verifier, and the registered client-key registry are owned by SDD-08 and SDD-10 and are reused unchanged by the revocation handler so that the verification path is identical to the claim handler's verification path. This chunk depends on, but does not implement, those primitives.
- The in-memory vault store and its concurrent-safe `Get` method are owned by SDD-03 and are available to the secret-retrieval handler. SIGHUP reload semantics (atomic vault swap, in-flight requests complete against the old vault) are owned by SDD-10.
- The audit-signing key is one of the keys derivable from the BIP32 hierarchy (per Constitution Principle III: `m/44'/7743'/2'`) and is loaded into the writer at server startup. This chunk does not own key derivation; it owns the writer's use of the loaded key.
- The genesis predecessor hash for the audit chain is a fixed, project-documented value (a deterministic constant such as a domain-separated all-zeros, all-ones, or a hash of a known string); the plan phase chooses the exact value subject to the constitution's hash requirements.
- The chat-platform mirror, when enabled, uses the same chat-platform connection the claim handler uses for approvals (Discord, per the project's current chat-platform choice); the mirror does not open an additional connection. The mirror's posting strategy (rate limiting, batching, summarisation) is a plan-phase choice; the spec only requires that the strategy not violate the best-effort discipline and the redaction rules in this document.
- Configuration fields needed by this chunk are present in the server's config schema (per `docs/CONFIG-SCHEMA.md`): the audit log file path (`audit_log`), the optional Discord audit channel identifier (`discord_audit_channel_id`), and any clock-sync threshold the health endpoint reports against. SDD-06 owns the config surface; if any field is not yet present there, the plan phase coordinates the addition before this chunk's implementation lands.
- The HTTP layer (router, middleware, request-id assignment, structured logging, panic recovery, graceful shutdown) is owned by SDD-10. This chunk's handlers are registered through the server's already-defined registration entry point and inherit middleware behaviour.
- The crash-vs-graceful-shutdown distinction is real: a graceful shutdown must drain the writer's buffer (User Story 8 / FR-039) but a hard crash (`SIGKILL`, kernel panic, power loss) may lose events that had been accepted into the writer's buffer but not yet persisted; this loss is documented as residual risk and is consistent with the constitution's audit invariants (which require tamper-evidence under attack, not durability under uncontrolled crash).
- "Request identifier" in handler responses and audit events refers to the server-side request identifier carried through the middleware chain, identical to the value used by the claim handler in the previous chunk; the same redaction rule applies (the request identifier is the only client-visible correlation handle).
- Tests for this chunk run against in-process fakes for the token validator/store, the vault store, the ECIES primitive, the audit-signing key, and the chat-platform mirror; they never open a real chat-platform connection, never write to a real disk-backed audit log outside a test temp directory, and never read a real session token from a real client.
- The chunk does not introduce new public CLI commands. The `hush revoke` CLI surface is owned by SDD-23. The endpoints in this chunk are HTTP-only and reachable through the random-prefix mount established by SDD-10.

## Out of scope

The following are explicitly out of scope for this chunk and are owned elsewhere:

- The CLI surface that calls these endpoints (`hush request`, `hush supervise`, `hush client status`, `hush revoke`) (SDD-16, SDD-23).
- The cryptographic primitives the handlers and audit writer depend on: BIP32 derivation (SDD-01), `SecureBytes` (SDD-02), vault file format (SDD-03), JWT issue/validate (SDD-07), canonical-JSON request signing (SDD-08), ECIES (SDD-09).
- The chat-platform integration (Discord bot, approver, rate limiter, disconnect monitor) (SDD-11).
- The claim handler that owns claim-pipeline outcomes the audit log records (SDD-12).
- The eight Discord alert classes (`[STALE]`, `[DAEMON]`, server-unreachable, etc.) (SDD-28). The audit mirror in this chunk publishes audit events as a separate concern from operator alerts.
- The supervisor-side state machine, status socket, validators, and watchdog (SDD-18 through SDD-22, SDD-26, SDD-27).
- The integration harness that exercises end-to-end lifecycle scenarios (SDD-25).
- Audit log rotation, archival, and disk-space management policies. Rotation is mentioned in the project's security documentation as a future concern; for this chunk the chain is append-only and ungated by size, with the assumption that operator monitoring catches disk-space pressure before it becomes a write failure.
- TLS within the trust mesh. Constitution Principle VI states the mesh is the transport-security boundary for v0.1.0; adding an inner TLS layer is a future amendment, not this chunk.
